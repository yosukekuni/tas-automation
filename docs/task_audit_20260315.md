## 6. 構造的課題

### 6-1. 「設計書」タスクの二重登録パターン
設計書を書いた際に「設計書作成」という別タスクが自動的に作られている形跡がある。
これにより設計完了54件のうち約13件が実質的な重複。
**対策**: 設計書作成はタスク内の成果物欄に記録し、別タスクを作らない運用ルールを徹底する。

### 6-2. 優先度の矛盾
「設計書」タスクが全て優先度「高」で作られているが、元タスクは「低」「中」。
テーブル内で「高優先度の設計完了」が大量に見えるため、本当に急ぐべきタスクが埋もれる。
**対策**: 設計書タスク統合後、元タスクの優先度を適切に再設定する。

### 6-3. 完了日の未入力（19件）
CLAUDE.md Rule 9に反する。タスク処理スクリプト（task_processor.py）に
「完了ステータス変更時に自動で完了日を記録する」ロジックを追加すべき。

### 6-4. 設計完了タスクの滞留
設計完了54件のうち、真に実装待ちのタスクは約35件。
うち売上直結のものは5件程度。残りはインフラ改善や個人タスクで、
「生存最優先」の運用方針に照らせば大半は凍結が妥当。
