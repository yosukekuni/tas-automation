## 1. 現状分析

### サイト基本情報
- URL: https://tomoshi.jp/
- ホスティング: GitHub Pages（静的HTML）
- リポジトリ: https://github.com/yosukekuni/tomoshi-site
- デプロイ: git push → GitHub Actions自動デプロイ
- **WordPressではない** → wp_safe_deploy.py は使用不可。直接HTMLを編集しgit push。

### 現状のCTA（全て高障壁）
| 配置場所 | テキスト | リンク先 | 障壁レベル |
|---------|---------|---------|-----------|
| ヒーロー | 「無料相談を予約する」 | Lark Scheduler | 高 |
| ヒーロー | 「050-7117-7141」 | tel:05071177141 | 高 |
| 固定ボトムバー | 「電話する」「無料予約」 | 同上 | 高 |
| フッター前 | 同上 | 同上 | 高 |

### 課題
- CTAが「面談予約」と「電話」のみ → 全て高障壁
- 「ちょっと興味がある」段階の訪問者がアクションできない
- リード獲得の入口がなく、ナーチャリング（ai_valueup_nurture.py）に流せない
- 面談予約の心理ハードルは非常に高い（経営者は「売り込まれる」警戒が強い）

## 2. 改善設計: 段階的CTA構造

### 2.1 低障壁CTA（接点確保 → リード獲得）

#### A. AI活用度セルフ診断チェックリスト
- **形式**: ページ内インタラクティブチェックリスト（JavaScript）
- **内容**: 10問程度の簡易診断
  - 「売上の50%以上が社長の人脈に依存している」
  - 「社長が1週間不在だと業務が止まる部門がある」
  - 「顧客情報は社長の頭の中にある」
  - 「見積もりは社長しか作れない」
  - 「月次決算に2週間以上かかる」等
- **結果表示**: スコア + 簡易アドバイス
- **リード獲得**: 「詳細レポートをメールで受け取る」→ メールアドレス入力
- **配置場所**: トップページ中段 + 専用ページ `/diagnosis/`
- **CTA文言**: 「無料で診断する（3分）」

#### B. 資料ダウンロード（PDF）
- **コンテンツ案**:
  - 「事業承継の5つの選択肢 - 廃業以外の道」（既存ブログ記事をPDF化）
  - 「AI活用で変わる中小企業の事業価値」
- **リード獲得**: メールアドレス入力でダウンロードリンク送信
- **配置場所**: ブログ記事内CTA + トップページサイドバー的位置
- **CTA文言**: 「無料ガイドを受け取る」

#### C. メールマガジン登録
- **内容**: 事業承継・AI活用の最新事例を月2回配信
- **リード獲得**: メールアドレスのみ
- **配置場所**: フッター上 + ブログ記事末尾
- **CTA文言**: 「事業承継の最新情報を受け取る」

### 2.2 中障壁CTA（具体的検討段階）

#### D. 無料相談フォーム（メール）
- **形式**: サイト内フォーム（面談ではなくメール返信）
- **入力項目**: 会社名、名前、メール、電話（任意）、業種、従業員数、相談内容（自由記述）
- **配置場所**: 専用ページ `/contact/` + トップページCTAセクション
- **CTA文言**: 「まずはメールで相談する」
- **レスポンス**: 24時間以内に返信（自動確認メール + 手動フォロー）

#### E. 事例集ダウンロード
- **コンテンツ案**: 業種別の変革事例をまとめたPDF
  - 「製造業のAI活用事例」「税理士事務所のDX事例」等
  - 現状サイトにある6業種の情報を深掘り
- **リード獲得**: メール + 会社名 + 業種
- **CTA文言**: 「事例集を無料で見る」

### 2.3 高障壁CTA（意思決定段階）— 現状維持

#### F. 面談予約（既存）
- **リンク先**: Lark Scheduler（変更なし）
- **CTA文言**: 「無料相談を予約する」（変更なし）
- **電話**: 050-7117-7141（変更なし）

## 5. 技術実装方針

### フォーム処理（GitHub Pages = 静的サイト制約）
GitHub Pagesはサーバーサイド処理不可のため、以下の方式を検討:

| 方式 | メリット | デメリット |
|------|---------|-----------|
| **Formspree** | 無料枠あり、設定簡単 | 月50件まで無料 |
| **Google Forms埋め込み** | 無料、スプシ連携 | デザイン制約 |
| **Lark Base Webhook** | CRM直接連携 | 実装やや複雑 |
| **GitHub Actions + API** | 完全自動化 | 開発コスト |

**推奨**: Formspree（初期）→ Lark Base Webhook（成長後）
- Formspreeでメール受信 → 手動 or zapierでLark CRM登録
- リード増加後にLark Webhook直接連携に移行

### セルフ診断（JavaScript）
- 静的HTMLに直接JavaScript実装
- ライブラリ不要（Vanilla JS）
- 結果表示後にFormspreeでメール取得

### PDF生成
- 既存ブログ記事（6本）をベースにCanva or HTML→PDF変換
- GitHub Repoの `/assets/pdf/` に配置

## 8. 注意事項

- tomoshi.jpはGitHub Pages（静的HTML）であり、WordPressではない
- wp_safe_deploy.pyは使用不可。`tomoshi-site` リポジトリを直接編集してgit push
- CSSで `inherit!important` 禁止（メモリルール準拠）
- フォーム処理にはサードパーティサービス（Formspree等）が必要
- TOMOSHIとTASのブランドは分離維持（メモリルール準拠）
- 社外秘情報（顧客名、ランウェイ等）はサイトに掲載しない
