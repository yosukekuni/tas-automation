# 報酬明細 デザイン・共有・自動化 提案書

## 1. 現状分析

### 現在のCSVフォーマットの問題点

1月度の明細CSV（新美光様）を確認した結果、以下の課題がある。

| 課題 | 詳細 |
|------|------|
| 視認性 | CSV形式のため列幅が崩れ、金額の位置が揃わない |
| セクション区分 | I〜V の区分はあるが、視覚的な区切りがない |
| 備考・根拠 | 計算根拠が明細と同一ファイルに長文で混在し、読みにくい |
| 改ざん防止 | CSVは容易に編集可能 |
| 印刷対応 | 列幅・改ページが制御できない |

### 現在のデータ構造（活かせる点）

- セクション分け（基本報酬/内業/諸手当/経費精算/控除）は論理的に整理されている
- 日付・単価・数量・根拠が明記されており、そのままPDF化の入力データになる
- 源泉徴収の計算根拠が丁寧に記載されている（これは別セクションで残す価値あり）

## 2. 報酬明細 PDFデザイン提案

### 基本方針

- **名称**: 「報酬明細書」（業務委託のため「給与明細」ではない）
- **形式**: A4縦・1ページ完結・PDF出力
- **生成方法**: Python + ReportLab（日本語フォント対応）

### デザインポイント

| 要素 | 仕様 |

| 稼働日一覧 | コンパクトに1行表示（詳細は出張申請で確認可能） |
| フッター | 「本書は電子発行です」の注記 |

## 3. 共有方法の比較と推奨

### TASの状況整理

- 対象者: 2名（新美光・政木勇治）
- 新美: Lark利用可
- 政木: 外部委託、**Lark DM不可**、メールのみ
- 月1回の送信（頻度低い）

### 比較表

| 方法 | セキュリティ | 手間 | 新美対応 | 政木対応 | 推奨度 |
|------|------------|------|---------|---------|--------|
| メール添付（平文PDF） | △ | ◎ 最小 | ○ | ○ | △ |
| パスワード付きPDF + メール | ◎ | △ 毎月PW通知 | ○ | ○ | ○ |
| Lark Drive共有フォルダ | ○ | ○ 初期設定のみ | ○ | × DM不可 | × |
| Google Drive共有 | ○ | △ SA設定 | ○ | △ | △ |
| Lark Bot DM + メール併用 | ○ | △ 2経路管理 | ○ Bot | ○ メール | △ |
| **Gmail下書き（PDF添付）** | **○** | **◎ 自動化済** | **○** | **○** | **◎** |

### 推奨: Gmail下書き自動作成（PDF添付）

**理由:**

1. **統一経路**: 2名とも同じメール送信で対応可能（政木のLark制約を回避）
2. **既存インフラ**: Gmail API（下書き作成）は既にTASで稼働中
3. **確認可能**: 下書きなので送信前に國本さんが目視確認できる（全自動だが最終承認あり）
4. **セキュリティ**: PDFは改ざん困難、メールはTLS暗号化
5. **手間ゼロ**: パスワード管理不要、フォルダ権限設定不要

**パスワード付きPDFは不採用の理由:**
- 月2通のためにパスワード管理の手間が見合わない
- パスワードを別経路で通知する手間（Lark/SMS）が発生
- 報酬明細の機密度は銀行口座番号等と比べて低い（金額のみ）

## 4. 自動化フロー提案

### 全体フロー

```
[毎月1日〜5日] 手動: CSVで稼働データ確定
        ↓
[スクリプト実行] generate_payslip.py
        ↓
  ┌─────────────────────────────────┐
  │ 1. CSVデータ読み込み             │
  │ 2. 報酬計算・源泉徴収計算       │
  │ 3. PDF生成（ReportLab）         │
  │ 4. Gmail下書き作成（PDF添付）    │
  │ 5. Lark タスクBaseに記録        │
  └─────────────────────────────────┘
        ↓
[國本さん] Gmail下書き確認 → 送信
        ↓
[振込] freee or 銀行振込
```

### Phase分け

#### Phase 1: PDF生成スクリプト（即時着手可能）

**ファイル**: `scripts/generate_payslip.py`

```
入力: CSVファイル（現行フォーマット）
出力: PDF（上記デザイン）
依存: reportlab, python標準ライブラリ
```

処理内容:
1. CSVパース（セクションI〜V を構造化）
2. 金額計算の検証（合計チェック）
3. ReportLabでPDF生成（A4縦、日本語フォント: IPAexGothic）
4. ファイル名: `報酬明細_新美光_202601.pdf`

#### Phase 2: Gmail下書き自動作成

**既存の仕組みを流用**:
- `gmail_create_draft` MCP（HTML形式）
- 添付ファイルはBase64エンコードでAPI送信

メール本文テンプレート:
```
件名: 【東海エアサービス】2026年1月度 報酬明細書

新美 光 様

お疲れ様です。東海エアサービスの國本です。
2026年1月度の報酬明細書を添付いたします。

■ 差引支払額: ¥170,415
■ 支払予定日: 2026年2月16日

ご不明点がございましたらお知らせください。

東海エアサービス株式会社
國本 洋輔
```

#### Phase 3: 稼働データ入力の自動化（将来）

現在CSVは手動作成だが、以下のデータソースから自動集計を検討可能:
- Lark CRM: 出張申請・現場記録
- Google Calendar: 稼働日
- 出張精算データ: 走行距離・高速代

これはPhase 1-2が安定稼働した後に検討する。

### 技術仕様

| 項目 | 仕様 |
|------|------|
| 言語 | Python 3.x |
| PDF生成 | ReportLab（pip install reportlab） |
| 日本語フォント | IPAexGothic（フリー、商用利用可） |
| Gmail API | 既存MCP経由 or gmail_utils.py |
| 出力先 | `/mnt/c/Users/USER/Documents/_data/payslips/YYYYMM/` |
| ファイル命名 | `報酬明細_{氏名}_{YYMM}.pdf` |

### GitHub Actions 連携（オプション）

月次実行のワークフローとして登録可能だが、入力CSVの確定タイミングが手動のため、当面は**ローカル実行（Claude Code経由）**が現実的。

```
運用イメージ:
1. 國本さん「○月分の明細作って」
2. Claude Code: CSVを読み込み → PDF生成 → Gmail下書き作成
3. 國本さん: Gmailで確認 → 送信
```
