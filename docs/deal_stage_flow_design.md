### 商談ステージ一覧（CRM上の値）

既存コードから判明しているステージ:
- (未設定) / 初回訪問 / 見積提出 / **受注** / **失注** / 不在 / 撮影完了 / 納品完了 / 請求済み / 完了（入金済み）

### 現状の問題

1. **ステージ変更の検知手段がない**: crm_monitorは新規レコード追加のみ検知。既存レコードのフィールド変更（ステージ遷移）は検知していない。
2. **受注時**: 受注台帳への転記は手動。サンクスメールもなし。次アクション日の運用ルールなし。
3. **失注時**: 即座にリサイクル対象になるが、cooldown期間の開始点管理なし。温度感のリセットなし。

## 2. 受注時フロー設計

### 2.1 トリガー条件

**商談テーブル（tbl1rM86nAw9l3bP）のステージフィールドが「受注」に変更されたことを検知**

#### 実装方式: ステージスナップショット比較（推奨）

crm_monitorの15分間隔実行に**ステージ変更検知機能**を追加する。

```
[方式] stateファイルに全商談の現在ステージを保存 → 次回実行時に比較
  - crm_monitor_state.json に "stage_snapshot": { record_id: stage } を追加
  - 差分があれば「ステージ変更イベント」として処理
```

**なぜWebhookでなくポーリングか:**
- Lark Bitable WebhookはEvent APIで受信可能だが、GitHub Actions環境では常駐サーバーを持てない
- 15分間隔のポーリングで実用上十分（受注は営業時間内の操作で、15分遅延は許容範囲）

## 3. 失注時フロー設計

### 3.1 トリガー条件

**商談テーブルのステージが「失注」に変更されたことを検知**
（受注と同じステージスナップショット比較方式）

### 3.2 失注時 自動実行アクション一覧

| # | アクション | 優先度 | 実装先 |
|---|-----------|--------|-------|
| 1 | CEO通知（Lark DM） | 必須 | crm_monitor.py |
| 2 | 商談レコード更新（リサイクル準備） | 必須 | deal_lost_handler() |
| 3 | 担当営業通知（失注分析プロンプト付き） | 推奨 | 同上 |
| 4 | lead_nurturing対象登録 | 自動 | lead_nurturing.py既存ロジックで対応 |

#### アクション詳細

**B1. CEO通知（Lark DM）**
```
失注確定
商談: {商談名}
取引先: {取引先名}
想定金額: {受注予定金額}
担当: {担当営業}
商材: {商材}
温度感: {温度感スコア}
→ {リサイクル予定日} に再アプローチ候補としてリスト化
```

**B2. 商談レコード更新**

| フィールド | 更新値 | 理由 |
|-----------|-------|------|
| 温度感スコア | Cold | 失注=一旦Cold化 |
| 次アクション | 「リサイクル待ち」 | lead_nurturing.pyのセグメントに合致させる |
| 次アクション日 | 失注日+90日 | リサイクルcooldown（lead_nurturing.pyのlost_dealsセグメントと一致） |

**ポイント**:
- lead_nurturing.pyは既に `lost_deals` セグメント（失注後3-6ヶ月経過 → 再アプローチ）を持っている
- 次アクション日を90日後に設定することで、cooldown明けにquote_followupやlead_nurturingの対象に自動的に乗る
- 手動で「営業見込みなし」と判断された場合は、次アクションを変更しない（将来的にセグメント除外条件として利用）

**B3. 担当営業通知**

- 失注理由の記録を促すメッセージを送信
- 営業が失注理由を商談メモに記録 → 将来の分析に活用

```
失注確定
商談: {商談名}
取引先: {取引先名}

失注理由をCRMの備考欄に記録してください:
- 価格（他社比較？予算不足？）
- タイミング（今期は不要？）
- 競合（どこに決まった？）
- その他

→ 90日後にリサイクル候補として再リスト化予定
```

### 3.3 失注→リサイクル遷移図

```
[失注]
  ↓ (自動) 温度感=Cold, 次アクション日=+90日, CEO/営業通知
  ↓ (自動) lead_nurturing.py lost_dealsセグメントに分類される
  ↓ ... cooldown 90日 ...
[リサイクル候補]
  ↓ (自動) lead_nurturing.py が再アプローチメール案を生成 → Lark通知
  ↓ (手動) 営業が再アプローチ実施
  ↓ (手動) 反応あり → ステージを「初回訪問」等に戻す / 反応なし → 次サイクルへ
```

## 4. 実装方針: crm_monitorへの組み込み

### 4.1 新規関数: check_stage_transitions()

`lark_crm_monitor.py` に追加する新関数。

```
def check_stage_transitions():
    """
    商談テーブルの全レコードのステージを前回スナップショットと比較し、
    変更があった場合にイベントハンドラを呼び出す。
    """

    処理フロー:
    1. stateファイルから前回の stage_snapshot を読み込み
    2. 商談テーブルの全レコードを取得
    3. 各レコードのステージを比較
    4. 変更があれば:
       - "受注" への変更 → handle_deal_won(record)
       - "失注" への変更 → handle_deal_lost(record)
       - その他の変更 → ログ出力のみ（将来拡張可能）
    5. 新しいスナップショットをstateに保存

    注意:
    - 初回実行時はスナップショット作成のみ（イベント発火なし）
    - fetch_all_dealsは他チェックと共用（API呼び出し節約）
```

### 4.2 crm_monitor_state.json の拡張

```json
{
  "record_counts": { ... },          // 既存
  "last_check": "...",               // 既存
  "stage_snapshot": {                 // 新規追加
    "recXXXXXX": "初回訪問",
    "recYYYYYY": "見積提出",
    ...
  },
  "stage_snapshot_updated": "2026-03-14T10:00:00"
}
```

### 4.3 mainへの組み込み

```
# Default実行（15分毎）に追加:
def main():
    ...
    # 既存チェック
    check_for_new_records()
    check_overdue_actions()
    check_hot_warm_no_action()
    check_new_deal_missing_fields()

    # 新規追加
    check_stage_transitions()     # ← ここに追加
    ...
```

### 4.4 handle_deal_won() の実装先

**選択肢A: crm_monitor.py内に実装**（推奨）
- メリット: 1回のAPI呼び出しで商談・取引先・連絡先データを共用できる
- crm_monitorは既にfetch_all_dealsを持つ。受注台帳への書き込みもLark API呼び出し1回
- 受注サンクスメールの部分だけdeal_thankyou_email.pyに委譲

**選択肢B: 別スクリプト deal_stage_handler.py に分離**
- メリット: 単体テストしやすい
- デメリット: API呼び出し回数が増える

→ **Aを推奨**。crm_monitorのfetch_all_dealsの結果を引数で渡す形で実装する。

### 4.5 GitHub Actions ワークフロー変更

crm_monitor.yml は変更不要（既にデフォルト実行で全チェックが走る構成）。
check_stage_transitions() をmainのデフォルトフローに追加するだけ。

### 4.6 受注サンクスメール: deal_thankyou_email.py の拡張

現在は「新規商談検出 → サンクスメール」のロジック。受注時メールを追加する場合:

- `--won-queue` オプションを追加
- crm_monitorのhandle_deal_won()から「受注メールキュー追加」APIを呼ぶか、
  あるいはキューファイル (thankyou_queue.json) に `type: "won"` のエントリを追加
- メールテンプレートを受注用に切り替え（Claude APIプロンプト分岐）

## 5. データフロー全体図

```
[営業が商談ステージを「受注」に変更]
    ↓
[crm_monitor.py（15分毎）]
    ↓ check_stage_transitions()
    ↓ stage_snapshotと比較 → 変更検知
    ↓
    ├─→ handle_deal_won()
    │     ├─ CEO Lark DM通知
    │     ├─ 受注台帳に自動レコード作成（Lark Bitable API）
    │     ├─ 商談レコード更新（次アクション日=+7営業日, 次アクション=納品準備）
    │     ├─ 担当営業通知（Lark DM or メール）
    │     └─ 受注サンクスメール → thankyou_queue.json に追加
    │
    └─→ handle_deal_lost()
          ├─ CEO Lark DM通知
          ├─ 商談レコード更新（温度感=Cold, 次アクション日=+90日, 次アクション=リサイクル待ち）
          ├─ 担当営業通知（失注理由記録依頼）
          └─ (lead_nurturing.pyが90日後に自動ピックアップ)
```

## 6. セーフガード

| リスク | 対策 |
|-------|------|
| 同一商談で重複実行 | stateのstage_snapshotで前回ステージを記録。同じ遷移は2度発火しない |
| API障害で受注台帳作成失敗 | CEOにエラー通知。次回実行時にリトライ（スナップショット更新しない） |
| 誤って受注ステージに変更 | 受注台帳レコード作成前に商談名で重複チェック。手動削除は可能 |
| 15分の検知遅延 | 受注/失注は即時性不要。通知が15分遅れても業務影響なし |
| サンクスメール誤送信 | 既存のreview_agent + 30日重複ウィンドウ + CEO事前プレビュー通知 |

## 8. テスト計画

1. **ドライラン**: `--dry-run` フラグで通知送信せずコンソール出力のみ確認

5. **テストレコード削除**: テスト完了後、テスト商談・受注台帳レコードを削除
